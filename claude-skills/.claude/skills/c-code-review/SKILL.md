# C / C++ Code Review

审查 C / C++ 相关代码时，优先关注内存安全、未定义行为、资源生命周期、ABI/API 兼容性和并发安全。不要为纯格式偏好阻塞合入，除非会影响可维护性或一致性。

## 优先级

1. **内存安全**：越界、悬垂指针、use-after-free、double free、泄漏、未初始化读。
2. **未定义行为**：整数溢出、错误类型转换、对齐、别名、生命周期、格式化字符串不匹配。
3. **资源管理**：文件描述符、socket、锁、内存、句柄、错误路径释放。
4. **接口兼容性**：头文件、结构体布局、枚举值、ABI、调用约定、二进制兼容。
5. **并发和安全**：数据竞争、锁顺序、死锁、输入校验、注入、敏感信息。

## 内存与生命周期

- 检查数组、缓冲区、字符串拷贝、长度计算是否越界。
- `malloc` / `calloc` / `realloc` 结果要检查；`realloc` 失败不能丢失原指针。
- 成功路径和所有错误路径都要释放资源。
- 指针释放后避免继续使用；必要时置空。
- 返回指针时确认指向对象生命周期足够长。
- C++ 代码优先 RAII 和智能指针，避免裸 `new` / `delete`。

## 字符串与缓冲区

- 避免不安全函数，如 `strcpy`、`strcat`、`sprintf`、`gets`。
- 使用带长度限制的 API 时仍要检查截断和结尾 `\0`。
- 格式化字符串必须匹配参数类型，尤其是 `%zu`、`%lld`、指针和平台相关类型。
- 输入长度必须校验，避免整数溢出后分配过小缓冲区。

## 未定义行为与可移植性

- 检查 signed integer overflow、移位位数、负数移位。
- 检查类型转换、指针对齐、strict aliasing 和大小端假设。
- 不要依赖未指定求值顺序。
- 平台相关代码要明确条件编译和类型宽度。
- 公共头文件应避免污染命名空间和宏冲突。

## 错误处理与资源

- 系统调用和库函数返回值必须检查。
- errno 使用要紧跟失败调用读取。
- 锁、文件、内存、句柄在所有错误路径都要释放。
- C 中建议使用单一 cleanup 出口统一释放；C++ 中优先 RAII。
- 失败日志应有上下文，但不能泄露敏感信息。

## 并发

- 共享状态必须有一致的锁或原子保护。
- 检查锁顺序是否可能死锁。
- 条件变量使用要配合循环检查条件。
- 原子变量要考虑内存序，避免过度使用 relaxed。
- 线程生命周期、join/detach、取消和退出路径必须清晰。

## 安全

- 外部输入进入路径、命令、SQL、解析器前必须校验。
- 避免命令字符串拼接；参数应分离。
- 文件路径要防目录穿越和 symlink 攻击。
- 解析二进制/网络协议时要检查长度、版本、边界和整数溢出。
- 日志不能输出密钥、token、密码或敏感 payload。

## API / ABI 兼容性

- 公共结构体字段、枚举值、函数签名、导出符号变化要评估兼容性。
- 头文件变更要检查所有调用方。
- C++ 虚函数、类布局、模板、inline 函数变更可能影响 ABI。
- 配置宏和编译选项变更要评估不同平台。

## 性能与可维护性

- 热路径关注不必要拷贝、重复分配、锁粒度、缓存局部性。
- 不要为了微优化牺牲安全性和可读性。
- 命名、模块边界、错误码、注释应与项目风格一致。
- 复杂宏要谨慎，优先清晰函数或类型安全封装。

## 测试

- 边界长度、错误路径、资源释放、并发场景需要测试。
- 内存相关改动建议配合 ASan/UBSan/TSan 或 Valgrind。
- 协议/解析器类代码建议有 fuzz 或异常输入测试。

## 输出要求

不要输出本 skill 的英文模板。将 C / C++ 发现合并到 `git-review` 的中文结构中：

- 内存安全、未定义行为、安全漏洞、ABI/API 破坏使用 `[严重]`
- 风格、局部可维护性、测试补强使用 `[建议]`
- 指明文件、函数、结构体或相关代码片段
- 给出具体、可操作的修复方向
